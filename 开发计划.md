# 文字奇幻剧情冒险游戏开发计划

> 目标：把“界隙书房”从一个有氛围的大厅原型，逐步发展成拥有状态、剧情分支、存档和完整移动端体验的文字奇幻冒险游戏。

## 开发原则

- `MainActivity.kt` 只负责 Android 生命周期、窗口设置和应用入口。
- 游戏状态、剧情数据、页面、组件、主题和存储必须分层组织。
- 先跑通一条完整的“大厅 → 副本 → 选择 → 返回大厅”闭环，再扩展内容。
- 所有会影响剧情的玩家操作都必须进入统一的游戏状态，而不是散落在 Composable 内部。
- 视觉动效服务于叙事状态，不为了装饰而堆叠动画。

## 1. 建立游戏状态模型与模块化结构

**目标：** 为所有后续页面建立唯一、可复用的游戏状态来源，并彻底避免把代码堆在 `MainActivity.kt`。

**内容：**

- 建立 `GameState`，包含章节、倒计时、当前记忆、剧情入口、余响、遗物、誓言和精神状态。
- 建立 `GameAction` 与状态 reducer，让用户操作通过明确动作更新状态。
- 将大厅页面拆分为页面、场景、组件和面板模块。
- 将静态大厅文案迁移到状态初始值，确保 UI 不再负责保存游戏数据。
- 为状态转换添加单元测试。

**完成标准：**

- `MainActivity.kt` 只保留入口代码。
- 大厅从 `GameState` 读取展示数据。
- 点击遗物、阅读余响、守住誓言都能更新状态。
- 状态测试覆盖初始数据和三个核心动作。

**当前状态：已完成**

## 2. 制作真正的剧情副本阅读页

**目标：** 让“雾中的无名门”打开一个可阅读、可继续的第三幕剧情页面。

**内容：**

- 增加应用内页面状态或导航边界。
- 实现剧情标题、场景文字、角色对白和段落推进。
- 保留当前大厅的仪式化进入动画。
- 支持系统返回键安全回到大厅。

**完成标准：**

- 点击无名门后进入第三幕剧情页。
- 玩家可以逐段阅读文本。
- 阅读结束后出现至少一个可操作选择。
- 返回大厅时不丢失当前游戏状态。

**当前状态：已完成**

新增文件：
- `app/src/main/java/io/application/game/story/Act3.kt` — 第三幕九段剧情、两个选择及结果文本
- `app/src/main/java/io/application/ui/story/StoryScreen.kt` — 逐段阅读页，含选择面板和结果展示

修改文件：
- `GameState.kt` — 新增 `StoryGateState.isVisited`、`GameAction.MakeChoice` 及 reducer 分支
- `GameApp.kt` — 新增 `AppScreen` 导航，`AnimatedContent` 切换大厅/副本
- `HallScreen.kt` — 新增 `onEnterStory` 回调，点击无名门后动画结束跳转副本

## 3. 实现选择分支系统

**目标：** 让玩家的选择真正改变剧情结果，而不是只有视觉上的按钮。

**内容：**

- 抽象 `StoryNode`、`StoryChoice` 和节点跳转关系。
- 为至少一个场景制作两个不同选择结果。
- 让选择可以添加余响、改变誓言、消耗遗物或改变精神状态。
- 对无效节点和缺失跳转进行显式错误处理。

**完成标准：**

- 同一剧情节点至少有两条可到达的结果路径。
- 每条路径都会产生可观察的状态差异。
- 重新进入大厅后能够看到选择留下的痕迹。

**当前状态：已完成**

新增文件：
- `game/story/StoryModel.kt` — `StoryScene`、`StoryChoice`（含 `resultParagraphs`）、`StoryEffect` 密封接口
- `game/story/Act3.kt` — 重写为 `act3Scene`，两条选择路径各含多段叙事段落和完整 effect 列表
  - "推开门"：`MarkGateVisited + AddEcho + ConsumeSelectedRelic`
  - "退后观察"：`MarkGateVisited + SetMentalState(HAUNTED) + AddEcho`

修改文件：
- `GameState.kt` — `MakeChoice` reducer 改为遍历 `StoryEffect` 列表（fold），unknown choiceId 静默 no-op
- `StoryScreen.kt` — `ResultSection` 接收 `List<StoryParagraph>` 展示分段叙事，`selectedChoice` 替代 `choiceResult`

## 4. 让大厅根据选择改变场景

**目标：** 实现"大厅会记住玩家"的核心体验。

**内容：**

- 中央记忆文案根据关键选择变化。
- 余响数量、内容和未读状态来自真实剧情结果。
- 无名门标题、颜色、描述或形状根据精神状态变化。
- 誓言和遗物状态影响大厅可见信息。
- 将大厅中的硬编码展示值全部替换为状态派生值。

**完成标准：**

- 至少三种不同选择会导致大厅出现不同反馈。
- 玩家能够从大厅判断自己过去做过什么。
- 状态变化同时体现在文字和至少一种视觉表现上。

**当前状态：已完成**

修改文件：
- `GameState.kt` — 新增 `displayMemoryPhrase`、`displayMemoryCaption`、`displayGate` 三个派生计算属性
  - 穿越门：记忆文案 → "我穿过了门 / 光是冷的"，门标题 → "你曾踏过的门"
  - 驻足观望：记忆文案 → "门后的声音 / 和我一模一样"，门标题 → "仍在等你的门"，精神状态 HAUNTED
- `HallScene.kt` — `RoomScene` 新增 `mentalState` 参数；光环颜色通过 `animateColorAsState` 平滑过渡（STEADY=琥珀，HAUNTED=紫，WOUNDED=暗红）
- `HallScreen.kt` — `RoomScene` 和 `StoryGate` 改用派生值（`displayMemoryPhrase`、`displayGate` 等）

## 5. 加入本地存档

**目标：** 让玩家关闭应用后仍能继续自己的故事。

**内容：**

- 使用 Android `DataStore` 保存可序列化的游戏状态。
- 建立状态版本和迁移策略，避免后续字段变化导致旧存档崩溃。
- 应用启动时恢复存档，没有存档时使用初始状态。
- 增加清除存档入口并要求二次确认。

**完成标准：**

- 关闭并重新打开应用后，章节、选择、余响、遗物和誓言都能恢复。
- 存档写入失败时有明确反馈。
- 清除存档不会留下旧的剧情状态。

## 6. 将剧情内容从 UI 代码中拆出

**目标：** 让新增章节不需要修改 Compose 页面。

**内容：**

- 建立 `story` 包，拆分剧情节点和选择数据。
- 建立 `StoryRepository`，负责按节点 ID 读取剧情。
- UI 只渲染剧情模型，不直接持有完整剧情文本。
- 为剧情节点、选择跳转和缺失数据添加测试。

**完成标准：**

- 新增一个剧情节点只需要添加数据，不需要修改页面布局代码。
- 剧情页面和大厅都通过统一仓库读取内容。
- 节点关系可以被测试验证。

## 7. 完善行囊与誓言系统

**目标：** 让大厅中的准备行为真正影响副本。

**内容：**

- 遗物支持选中、带入、消耗和剧情条件判断。
- 限制一次进入副本可以携带的遗物数量。
- 誓言支持确认、改变和剧情条件判断。
- 在剧情页中根据遗物和誓言解锁或隐藏选择。

**完成标准：**

- 玩家选择的遗物会影响至少一个剧情节点。
- 誓言会改变至少一条对白或选择。
- 遗物消耗和誓言变化会正确反映到大厅。

## 8. 完善移动端体验与可访问性

**目标：** 让沉浸感在不同设备和系统设置下仍然可靠。

**内容：**

- 适配小屏、大屏、横屏和系统字体放大。
- 检查所有触控区域、文本对比度和焦点顺序。
- 正确处理系统返回键、面板关闭和剧情退出。
- 为动画提供减少动效或低动效模式。
- 修正“长按进入”文案与真实手势的一致性。

**完成标准：**

- 主要操作在小屏设备上不遮挡、不溢出。
- 所有面板和剧情页面都有明确返回路径。
- 动效关闭后功能仍然完整可用。

## 9. 加入音效与触感反馈

**目标：** 用声音和触感强化文字选择的仪式感。

**内容：**

- 为开门、翻页、阅读余响、选择遗物和确认誓言准备音效接口。
- 为关键交互增加轻微触感反馈。
- 将音效和触感偏好放入设置。
- 不让音效阻塞剧情阅读，也不在无障碍模式中强制播放。

**完成标准：**

- 关键交互有一致的声音或触感反馈。
- 用户可以关闭音效和触感。
- 没有音频资源时，核心剧情仍可完整运行。

## 10. 测试、性能与发布准备

**目标：** 在扩展剧情内容前确保体验稳定。

**内容：**

- 增加状态 reducer、剧情跳转、存档恢复和关键 UI 行为测试。
- 检查动画重组、长文本性能和内存占用。
- 使用不同屏幕尺寸进行截图回归。
- 检查 release 构建、签名配置和备份策略。
- 完善用户可见的运行说明与版本变更记录。

**完成标准：**

- 核心游戏循环有自动化测试。
- 主要页面在目标设备上没有明显溢出或卡顿。
- 可以生成可安装的 release 包，并明确发布前仍需的配置。

## 推荐执行顺序

```text
1 状态模型与模块化
→ 2 剧情副本页
→ 3 选择分支
→ 4 大厅状态反馈
→ 5 本地存档
→ 6 剧情数据化
→ 7 行囊与誓言
→ 8 移动端与无障碍
→ 9 音效与触感
→ 10 测试与发布
```
